# PC03 — Diseño detallado, contratos, integración, resiliencia y observabilidad

Esta PC03 evalúa la capacidad de diseñar, justificar y evidenciar una arquitectura distribuida para el caso **Tutorías Universitarias**. No se calificará únicamente que el código “corra”; se calificará que las decisiones arquitectónicas estén claras, que los contratos sean verificables y que exista evidencia razonable de integración, resiliencia, seguridad y observabilidad.

## Antes de empezar: comprensión mínima del proyecto

Esta evaluación usa como caso base el proyecto académico **Tutorías Universitarias**. Antes de responder, el estudiante debe comprender qué problema resuelve el sistema, qué servicios participan, cómo se integran y qué evidencias existen en el repositorio. El objetivo de esta sección es evitar respuestas genéricas y asegurar que el análisis esté conectado con el caso real.

### Qué problema resuelve el sistema

El sistema modela una plataforma donde un estudiante solicita una tutoría con un tutor disponible. Para completar el flujo, la solución debe autenticar al usuario, validar participantes, revisar disponibilidad, bloquear agenda, registrar la tutoría, emitir una notificación y dejar trazabilidad observable del proceso.

El caso permite analizar problemas reales de arquitectura distribuida: límites entre servicios, contratos HTTP, mensajería, consistencia eventual, fallas parciales, compensación, resiliencia, observabilidad, seguridad y operación local.

### Servicios principales

| Servicio o componente | Responsabilidad principal |
| --- | --- |
| `client-mobile-sim` | Cliente web de simulación para solicitar tutorías. |
| `tracking-dashboard` | Dashboard de trazabilidad para observar eventos del flujo. |
| `ms-auth` | Emite tokens JWT para autenticación. |
| `ms-usuarios` | Expone información de estudiantes y tutores. |
| `ms-agenda` | Verifica disponibilidad y administra bloqueos de horario. |
| `ms-tutorias` | Orquesta la solicitud de tutoría y coordina validaciones, agenda y notificación. |
| `ms-notificaciones` | Procesa eventos de notificación mediante RabbitMQ. |
| PostgreSQL | Persistencia separada para usuarios, agenda y tutorías. |
| RabbitMQ | Broker de mensajería para integración asíncrona y notificaciones. |
| Toxiproxy | Inyección controlada de fallas para validar resiliencia. |
| Prometheus/Grafana | Métricas y visualización para observabilidad local. |

### Flujo principal de solicitud de tutoría

1. El cliente obtiene un token mediante `ms-auth`.
2. El cliente solicita una tutoría a `ms-tutorias` usando el token.
3. `ms-tutorias` valida identidad y rol del estudiante.
4. `ms-tutorias` consulta `ms-usuarios` para validar estudiante y tutor.
5. `ms-tutorias` consulta `ms-agenda` para revisar disponibilidad.
6. `ms-agenda` bloquea el horario si está disponible; si ya existe un bloqueo, responde conflicto `409`.
7. `ms-tutorias` registra la tutoría y publica un evento para notificación.
8. `ms-notificaciones` consume el evento desde RabbitMQ.
9. El dashboard y las métricas permiten observar el comportamiento del flujo y sus fallas.

### Integración síncrona y asíncrona

El caso combina dos estilos de integración:

- **Síncrona HTTP:** `ms-tutorias` llama a `ms-usuarios` y `ms-agenda` para validar datos y disponibilidad.
- **Asíncrona con RabbitMQ:** los eventos permiten desacoplar notificaciones y trazabilidad del flujo principal.

Esta combinación es deliberada: permite discutir cuándo conviene orquestar desde un servicio central, cuándo conviene publicar eventos, qué ocurre ante fallas parciales y cómo se evita que un servicio lento o caído arrastre a todo el sistema.

### Escenario: Pagos de Tutorías

El equipo debe incorporar la funcionalidad **Pagos de Tutorías** al sistema existente. El flujo actual permite solicitar una tutoría, validar usuario y tutor, bloquear agenda, registrar la tutoría y emitir notificaciones. La nueva funcionalidad exige que una tutoría pagada solo avance cuando exista una intención de pago válida y trazable.

El sistema deberá interactuar con un nuevo servicio `ms-pagos`, que a su vez se comunica con proveedores externos como Visa o Mastercard. Estos proveedores pueden responder lento, rechazar operaciones, aceptar una operación después de varios segundos, devolver respuestas duplicadas, enviar callbacks tardíos o no responder dentro del tiempo esperado por el cliente.

El diseño debe considerar como mínimo:

- duplicidad de solicitudes por doble clic, refresh del cliente, retry automático o pérdida de respuesta;
- demoras del proveedor de pagos superiores al timeout HTTP del cliente o del gateway;
- pagos aprobados después de que `ms-tutorias` canceló o abandonó la solicitud;
- fallas parciales entre bloqueo de agenda, registro de tutoría, autorización de pago y notificación;
- trazabilidad de extremo a extremo con identificadores de correlación;
- seguridad de datos sensibles de pago y separación de responsabilidades entre servicios;
- operación cloud con demanda variable y costos razonables.

### Diseño propuesto deficiente que debe analizar

Un equipo propone implementar Pagos de Tutorías de la siguiente manera:

1. `ms-tutorias` recibe la solicitud del cliente para reservar y pagar una tutoría.
2. `ms-tutorias` abre una transacción en su base de datos.
3. Dentro de esa transacción, `ms-tutorias` llama por HTTP a `ms-pagos`.
4. `ms-pagos` llama por HTTP al proveedor externo, por ejemplo Visa o Mastercard, y espera la respuesta final.
5. Si el proveedor responde `OK`, `ms-pagos` responde `OK` a `ms-tutorias`.
6. `ms-tutorias` guarda la tutoría como confirmada y cierra la transacción.
7. Si ocurre timeout, cancelación del cliente o error HTTP, `ms-tutorias` revierte la transacción y devuelve error.
8. Si el cliente reintenta la misma operación, se ejecuta todo el flujo nuevamente.

El equipo argumenta que este diseño es simple porque “todo ocurre en orden” y porque `ms-tutorias` conserva el control del proceso. Su tarea es evaluar rigurosamente esa afirmación.

### Errores frecuentes y qué no debe hacer

- No entregue un cuestionario de respuestas cortas; esta Parte A exige un RFC breve, defendible y ordenado.
- No afirme que el diseño es correcto solo porque “usa transacciones” o porque “todo ocurre en orden”. En sistemas distribuidos, el orden aparente no elimina fallas parciales.
- No proponga retries sin límite, sin timeout, sin backoff y sin idempotencia.
- No ignore respuestas tardías del proveedor de pagos, callbacks duplicados o estados ambiguos.
- No mezcle datos sensibles de tarjetas en servicios que no deberían conocerlos.
- No describa observabilidad como “guardar logs” sin explicar qué se correlaciona, qué métrica alerta y cómo se diagnostica un incidente.
- No agregue colas, eventos, DLQ, Service Mesh u otros patrones como decoración. Debe explicar qué riesgo reducen y qué costo agregan.
- No entregue diagramas decorativos que no muestren responsabilidades, dependencias, estados o fallas relevantes.
- No dependa de ejecutar el sistema local para responder. La Parte A es teórica y debe poder resolverse con el contexto del proyecto, contratos, documentación de apoyo y razonamiento arquitectónico.
- No incluya código completo, solucionarios, credenciales, tokens reales, secretos ni datos personales innecesarios.
